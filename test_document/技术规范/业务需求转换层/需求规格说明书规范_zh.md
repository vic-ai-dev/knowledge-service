# 需求规格说明书 / BRD→TRD 规范
> 参考：阿里巴巴需求管理实践、腾讯 TAPD 研发协同规范、华为 IPD 需求管理流程

## 1. 概述

### 1.1 目的
规范保险核心系统（核保、核赔、续期、保全等）需求从商业需求文档到技术需求文档的翻译过程，确保需求传递准确、可追溯、可验证。

### 1.2 适用范围
适用于保险公司技术开发部承接的所有业务需求，包括核心系统、渠道系统、数据平台等。

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| BRD | 商业需求文档（Business Requirements Document），由业务方输出 |
| TRD | 技术需求文档（Technical Requirements Document），由技术开发部输出 |
| 需求追溯矩阵 | 从 BRD 条目到 TRD 条目到测试用例的映射表 |
| 需求变更 | 已基线化的需求发生修改 |

---

## 2. 输入：BRD 要求

### 2.1 BRD 必备内容
业务方提交的 BRD 必须包含以下章节：

1. **需求背景与目标** — 业务痛点、目标用户、预期收益（量化指标）
2. **业务规则描述** — 核保规则/赔付规则/续期规则/保全规则（含公式、条件分支、例外情况）
3. **业务流程图** — BPMN 2.0 或 Swimlane 图
4. **数据需求** — 输入字段、输出字段、数据来源、历史数据迁移要求
5. **非功能需求** — 响应时间、处理量级、可用性要求、合规要求
6. **验收标准** — 可量化的验收条件

### 2.2 BRD 质量门禁

- [ ] 所有业务规则覆盖正常流与异常流
- [ ] 涉及账务的条目附带会计分录示例
- [ ] 涉及监管报送的条目附带监管报文样例
- [ ] 经业务负责人（签字/邮件）确认

---

## 3. BRD → TRD 翻译规范

### 3.1 翻译原则
每一条 BRD 条目必须翻译为 TRD 中的完整技术方案，不允许未经分析的直接照搬。

| BRD 条目示例 | TRD 翻译要求 |
|--------------|--------------|
| "续期保费不足时发送提醒" | 定义：判断时机（T+N日）、提醒渠道（短信/APP推送/邮件）、提醒频率（日频/单次）、失败重试策略 |
| "核保引擎支持加费承保" | 定义：加费计算规则表、因子表结构、加费上下限、与费率表的联动关系 |
| "保全收付费走新资金平台" | 定义：接口协议、报文映射表（核心字段→资金平台字段）、对账机制、异常处理流程 |

### 3.2 TRD 模板结构

```markdown
# TRD-[序号] [需求标题]

## 1. 需求概要
- 关联 BRD：[BRD编号]
- 关联需求来源：[Jira/工单编号]
- 涉及系统：[核心系统/渠道平台/资金平台/数据中台]

## 2. 需求分析
### 2.1 业务流程分析
### 2.2 影响分析
### 2.3 风险评估

## 3. 技术方案
### 3.1 方案描述
### 3.2 接口设计
### 3.3 数据库设计
### 3.4 业务规则实现
### 3.5 异常处理

## 4. 非功能需求实现
## 5. 验收标准映射
## 6. 需求追溯
## 7. 变更记录
```

## 4. 评审流程

### 4.1 评审节点
1. **TRD 初稿评审**（技术内部）— 架构师 + 开发负责人 + QA
2. **TRD 联合评审**（业务 + 技术）— 业务方确认业务规则翻译无误
3. **TRD 基线** — 评审通过后锁定，签入代码仓库

### 4.2 评审Checklist

- [ ] 每条 BRD 条目都有对应的 TRD 方案
- [ ] 技术方案有 A/B 选项时已做优劣分析
- [ ] 接口定义覆盖正常、超时、异常三种场景
- [ ] 数据库变更包含回滚方案
- [ ] 非功能需求已被评估并在方案中体现
- [ ] 变更影响范围明确标注

---

## 5. 需求变更管理

### 5.1 变更分级

| 级别 | 定义 | 审批链 |
|------|------|-------|
| 一级 | 界面文案/非功能调整，不影响接口与数据 | 技术负责人确认 |
| 二级 | 业务逻辑调整，涉及接口变更 | 技术负责人 + 业务方确认 |
| 三级 | 数据模型变更、流程重构、新增系统对接 | PMO + 架构评审委员会审批 |

### 5.2 变更流程
1. 提交变更申请（关联原始 BRD/TRD）
2. 影响评估（24h 内输出）
3. 变更评审
4. 更新文档基线
5. 通知受影响干系人
