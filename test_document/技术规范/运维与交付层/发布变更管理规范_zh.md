# 发布变更管理规范

> 参考：ITIL 变更管理最佳实践、腾讯变更管理规范、Google SRE 变更管理、阿里云变更管理

## 1. 概述

### 1.1 目的
规范技术开发部的变更管理流程，确保所有变更经过分级评估、合理审批、可控执行，降低变更引入生产风险的概率。

### 1.2 适用范围
适用于技术开发部所有对生产环境有影响的变更，包括但不限于：应用发布、数据库变更、配置修改、基础设施变更、第三方服务切换等。

### 1.3 术语定义

| 术语 | 定义 |
| --- | --- |
| 变更 | 对生产环境或预发布环境的任何修改操作 |
| CAB | 变更顾问委员会（Change Advisory Board） |
| 小黑窗 | 允许执行高风险变更的固定时间窗口 |
| RTO | 恢复时间目标（Recovery Time Objective） |
| RPO | 恢复点目标（Recovery Point Objective） |

## 2. 变更分级

### 2.1 变更类别

| 类别 | 定义 | 示例 |
| --- | --- | --- |
| 标准变更 | 预授权、低风险、执行流程固定的变更 | 日常配置修改、白名单添加 |
| 普通变更 | 需要评审和审批的中等风险变更 | 功能发布、数据库 DDL |
| 紧急变更 | 需要快速执行以修复故障的变更 | P0/P1 故障修复、安全补丁 |
| 重大变更 | 高风险、跨团队、影响广泛的变更 | 架构重构、核心系统替换、数据迁移 |

### 2.2 变更分级矩阵

| 级别 | 影响范围 | 故障风险 | 审批链 | 执行窗口 |
| --- | --- | --- | --- | --- |
| L1（低） | 单服务/单模块 | 低 | 技术负责人 | 工作日 |
| L2（中） | 多服务/跨模块 | 中 | 技术负责人 + 架构师 | 预发布窗口 |
| L3（高） | 跨系统/影响业务 | 高 | CAB 评审 | 小黑窗 |
| L4（重大） | 全平台/核心系统 | 极高 | CTO / VP 审批 + CAB | 小黑窗（特殊排期） |

### 2.3 标准变更清单

以下变更被预授权为标准变更，无需逐个审批但需记录：

- 监控告警阈值的调整
- 白名单 IP 的增删
- 日志级别调整（非生产环境）
- 配置中心上的非功能开关变更
- 只读用户的授权

## 3. 变更审批流程

### 3.1 普通变更流程

```
变更申请 > 影响评估 > 技术评审 > 审批 > 排期 > 执行 > 验证 > 关闭
```

| 步骤 | 负责方 | 输出 | 时限 |
| --- | --- | --- | --- |
| 1. 变更申请 | 变更发起人 | 变更单（描述、范围、方案、回滚） | — |
| 2. 影响评估 | 技术负责人 | 影响分析报告 | 2h |
| 3. 技术评审 | 架构师（L2+） | 评审意见 | 4h |
| 4. 审批 | 审批人（按分级） | 审批结论 | 4h |
| 5. 排期 | 技术负责人 | 执行排期 | — |
| 6. 执行 | 开发/运维 | 执行记录 | 按排期 |
| 7. 验证 | 发起人 + QA | 验证报告 | 执行后 30min |
| 8. 关闭 | 技术负责人 | 变更关闭 | 验证通过后 |

### 3.2 紧急变更流程

```
故障发现 > 紧急评估 > 紧急审批（可先执行后补）> 执行 > 验证 > 事后补单
```

- 紧急变更审批人：紧急变更评审组（技术负责人 + 架构师 + 值班 DBA）
- 紧急变更执行后 24 小时内必须补交变更单和复盘报告
- 同一服务一个月内出现 3 次以上紧急变更，需进行根因分析

### 3.3 重大变更流程

- 提前 1 周提交变更方案
- 需召开 CAB 会议评审
- 需进行全链路压测和灰度验证
- 需制定详细的回滚方案和应急预案
- 需提前通知所有受影响干系人
- 执行时需安排至少 2 人值班

## 4. 小黑窗制度

### 4.1 定义

小黑窗（Change Window）是指允许执行高风险变更的固定时间段，期间要求关键干系人全程在线。

### 4.2 小黑窗安排

| 项目 | 说明 |
| --- | --- |
| 常规小黑窗 | 每周四 22:00 - 次日 02:00 |
| 紧急小黑窗 | 按需申请，CTO 批准 |
| 冻结期 | 月底倒数 3 天 + 法定假期前 3 天（仅允许紧急变更） |
| 年度冻结 | 春节前 2 周至春节后 1 周（仅允许 P0 级修复） |

### 4.3 小黑窗纪律

- 小黑窗内变更需提前 2 小时提交审批
- 小黑窗开始后 30 分钟内未开始执行的变更，自动延期到下个窗口
- 小黑窗结束时未完成的变更，若处于可回滚状态则回滚
- 小黑窗禁止并行执行多个 L3 变更

## 6. 变更回顾

### 6.1 变更总结

每周五进行变更总结会议，统计：

- 本周变更总数
- 各级别变更数量
- 变更成功率
- 回滚变更数及原因
- 紧急变更数及根因

### 6.2 变更指标

| 指标 | 目标 | 告警阈值 |
| --- | --- | --- |
| 变更成功率 | > 99% | < 98% |
| 紧急变更占比 | < 10% | > 20% |
| 回滚率 | < 5% | > 10% |
| 变更导致事故数 | 0 | 1 |
| 变更平均前置时间 | < 3 天 | > 7 天 |
