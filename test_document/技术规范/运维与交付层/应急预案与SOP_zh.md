# 应急预案与SOP

> 参考：Google SRE 事件响应实践、阿里云故障管理 SOP、腾讯应急响应规范、ITIL 事件管理

## 1. 概述

### 1.1 目的
建立标准化的故障响应机制和处置流程，确保核心服务发生故障时能够快速定位、高效恢复、事后改进，最大限度降低业务损失。

### 1.2 适用范围
适用于技术开发部负责的所有系统在生产环境发生的故障和突发事件。

---

## 2. 故障分级

### 2.1 故障严重级别

| 级别 | 定义 | 示例 | 响应时间 | 升级时间 |
| --- | --- | --- | --- | --- |
| S0（灾难级） | 核心系统完全不可用，或发生资损 | 核心宕机、账务错误、数据丢失 | 立即 | 10min |
| S1（严重级） | 核心模块功能严重受损，影响面大 | 出单不可用、支付失败、大面积慢 | 5min | 30min |
| S2（普通级） | 非核心功能受损，业务有替代方案 | 报表导出失败、部分渠道异常 | 30min | 2h |
| S3（提醒级） | 轻微异常，无业务影响 | 单接口偶发超时、告警误报 | 24h | — |

### 2.2 资损故障特别说明

涉及资金损失的故障自动定为 S0，需同时：

1. 立即止损（停止相关交易/功能降级）
2. 通知财务部门
3. 保留全量日志和数据库快照
4. 法务部门介入评估损失和合规风险

## 3. 应急组织架构

### 3.1 角色定义

| 角色 | 人员 | 职责 |
| --- | --- | --- |
| 事件指挥官（IC） | 值班技术负责人 | 统筹故障处置，调配资源，对外沟通 |
| 一线响应 | 值班工程师 | 初步排查、执行处置措施 |
| 技术专家 | 相关模块负责人 | 深度排查、修复方案设计 |
| 沟通负责人 | 项目经理/PMO | 向业务方和管理层通报进展 |
| Scribe（记录员） | 值班人员/实习生 | 记录故障时间线和处置步骤 |

### 3.2 汇报线

```
一线响应发现故障
    ↓
事件指挥官（IC）确认故障级别
    ↓
S0/S1 → 立即建立应急群（技术+业务+管理层）
S2   → 通知技术负责人和相关模块Team
S3   → 记录工单，日间处理
    ↓
故障恢复 → 复盘（Postmortem）
```

## 4. 故障处置流程（SOP）

### 4.1 发现与响应

```
1. 告警触发 / 用户反馈 / 值班巡检发现故障
2. 确认故障：是否真实、影响范围、严重级别
3. IC 接管指挥权，建立应急通讯群
4. 在 5 分钟内完成初始公告（故障现象、影响范围、预计处理时间）
```

**初始公告模板：**

```
【故障通告】[系统名称] [故障级别]
故障现象：[故障描述]
开始时间：[时间]
影响范围：[受影响功能/用户量]
当前状态：[排查中/处置中/已恢复]
预计恢复时间：[时间]
处理人：[IC 姓名]
```

### 4.2 排查与定位

```
1. 查看监控大盘（Grafana / Kibana）
2. 查看近期变更记录（是否由变更引发）
3. 查看应用日志（ERROR 级别日志/错误堆栈）
4. 查看数据库状态（连接数/慢查询/主从延迟）
5. 查看基础设施（CPU/内存/磁盘/网络）
6. 查看外部依赖（第三方 API / 中间件状态）
```

**排查检查表：**

```
□ 是否有变更刚刚上线？（查看变更系统）
□ 数据库是否有慢查询或死锁？（查看慢查询日志）
□ 是否有大批量请求涌入？（查看 QPS 监控）
□ 外部依赖是否正常？（查看健康检查）
□ 是否有定时任务在运行？（查看调度中心）
□ 证书/密钥是否过期？（查看证书有效期）
```

### 4.3 处置与恢复

| 故障类型 | 处置措施 | 止损手段 |
| --- | --- | --- |
| 应用宕机 | 重启服务 / 回滚版本 | 切流到备用实例 |
| 数据库死锁 | Kill 阻塞会话 / 优化 SQL | 主备切换 |
| 慢查询打满资源 | Kill 慢查询 / 添加索引 | 限流 / 降级 |
| 中间件故障 | 重启 / 切换集群 | 降级到本地缓存 |
| 网络故障 | 切换 DNS / 切换可用区 | CDN / 多活切换 |
| 数据错误 | 根据备份恢复 / 数据订正 | 暂停相关交易 |
| 安全攻击 | 封禁 IP / 开启 WAF | 服务降级 / 切备用 |

**执行原则：**
- **止损优先**：先恢复服务，再排查根因
- **回滚为主**：如果是变更导致，优先回滚而非在线修复
- **一人操作一人复核**：高危操作需要双人确认
- **记录每一步**：Scribe 记录所有操作和时间点

### 4.4 恢复与确认

```
1. 执行恢复操作
2. 验证服务健康（/health + 核心接口冒烟）
3. 确认业务指标恢复（出单量/支付成功率回到正常水平）
4. 持续观察 30 分钟（无异常波动）
5. 解除告警，恢复通知
6. 关闭应急群（进入复盘阶段）
```

## 5. 典型场景处置SOP

### 5.1 核心服务宕机

```
适用：Policy Service / Underwriting Service 无响应

步骤：
1. 确认服务进程状态（ps / kubectl get pods）
2. 查看最近日志（最近 5 分钟的 ERROR 和异常堆栈）
3. 若是 OOM：增加内存配置，重启实例
4. 若是死锁/GC 问题：抓取 Thread Dump 和 Heap Dump
5. 若是网络问题：检查服务发现（Nacos/Eureka）和网络策略
6. 若是代码缺陷：回滚到上一个稳定版本
7. 恢复后：逐步加回流量，观察 30 分钟
```

### 5.2 数据不一致

```
适用：核心系统和资金平台对账不平、保单状态不一致

步骤：
1. 首先核对数据差异，确认影响范围
2. 评估是否需要暂停相关交易
3. 如果是账务不一致：切换到备用对账流程，保留原始凭证
4. 如果是状态不一致：通过补偿事务或数据订正脚本修复
5. 记录所有手动修正操作（who, what, when, why）
6. 修正后做全量对账验证
7. 复盘：补充对账监控和预防机制
```

### 5.3 数据库故障

```
适用：数据库连接池满、主从延迟过大、磁盘空间不足

主从延迟过大：
1. 确认从库复制进程状态（SHOW SLAVE STATUS）
2. 若延迟持续增长，排查大事务或慢查询
3. 紧急处理：跳过错误事务（需谨慎）或重建从库
4. 长期方案：优化大事务、升级从库配置

连接池满：
1. 查看活跃连接和等待连接
2. Kill 空闲连接或长时间未提交的事务
3. 检查应用是否连接泄漏（连接池监控）
4. 紧急扩容连接池上限

磁盘空间不足：
1. 清理归档日志（binlog / slow log）
2. 清理临时表空间
3. 紧急扩容磁盘
4. 定期清理策略检查
```

## 6. 故障复盘（Postmortem）

### 6.1 复盘原则

| 原则 | 说明 |
| --- | --- |
| 非惩罚性 | 复盘的目的是改进系统，而非追责 |
| 关注根因 | 找到根本原因，而非停留在表面现象 |
| 关注系统性缺陷 | 个人错误背后往往是流程/工具/培训的不足 |
| 可执行改进 | 每个 Action Item 必须有明确的负责人和截止时间 |

### 6.2 复盘模板

```markdown
# 故障复盘 - [故障编号]

## 基本信息
- 故障标题：[标题]
- 严重级别：[S0/S1/S2/S3]
- 时间范围：[开始] - [结束]
- 时长：[X] 分钟
- 影响范围：[受影响系统/功能/用户量]
- 资损金额：[如有]

## 故障时间线
| 时间 | 事件 | 操作人 |
| --- | --- | --- |
| [T0] | 故障发生 | 系统 |
| [T+5min] | 告警触发 | 系统 |
| [T+10min] | IC 确认 S1 级别 | [姓名] |
| [T+20min] | 发现根因：... | [姓名] |
| [T+30min] | 执行回滚 | [姓名] |
| [T+45min] | 服务恢复 | [姓名] |

## 根因分析
### 直接原因
[导致故障的触发条件]

### 根本原因
[- 代码缺陷：...
 - 缺乏测试覆盖：...
 - 缺少监控告警：...
 - 流程缺失：...]

## 处置评估
| 环节 | 评价 | 改进点 |
| --- | --- | --- |
| 发现速度 | 快/中/慢 | [改进建议] |
| 定位效率 | 高/中/低 | [改进建议] |
| 处置措施 | 有效/无效 | [改进建议] |
| 沟通质量 | 好/中/差 | [改进建议] |

## Action Items
| 序号 | 改进项 | 负责人 | 截止日期 |
| --- | --- | --- | --- |
| 1 | [改进项] | [姓名] | [日期] |
| 2 | [改进项] | [姓名] | [日期] |
```
