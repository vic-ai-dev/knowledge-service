# 性能压测规范

> 参考：阿里云 PTS 最佳实践、腾讯全链路压测方案、Google 容量规划实践、JMeter 官方指南

## 1. 概述

### 1.1 目的
规范保险核心系统的性能压测流程，确保系统在承保高峰期、促销活动等大流量场景下稳定运行，满足业务 SLA 要求。

### 1.2 适用范围
适用于技术开发部所有需要上线的新系统、大版本迭代、以及定期的容量评估压测。

---

## 2. 压测分类与频率

| 压测类型 | 触发条件 | 执行频率 |
| --- | --- | --- |
| 常规压测 | 大版本发布、架构调整 | 每个大版本前 |
| 峰值压测 | 承保高峰期前（如开门红、车险到期季） | 高峰前 1 个月 |
| 容量评估 | 常规容量规划、业务量增长超预期 | 每季度一次 |
| 稳定性压测 | 长时间运行验证 | 大版本前 |
| 异常压测 | 容灾演练、混沌工程 | 每半年一次 |

---

## 3. 性能指标定义

### 3.1 通用指标

| 指标 | 定义 | 目标值参考 |
| --- | --- | --- |
| TPS / QPS | 每秒事务/请求数 | 见 3.2 节 |
| RT（响应时间） | 请求从发起到响应的时间 | P50 < 500ms, P99 < 2000ms |
| 错误率 | 失败请求占比 | < 0.1%（核心接口） |
| 并发用户数 | 同时在线操作的用户数 | 按业务场景预估 |
| 吞吐量（MB/s） | 网络/磁盘吞吐 | 不超过带宽的 60% |

### 3.2 保险核心系统指标参考

| 业务场景 | 预估峰值 QPS | 目标 P99 RT | 数据来源 |
| --- | --- | --- | --- |
| 在线投保 | 500 | < 1000ms | 开门红期间峰值 |
| 保单查询 | 2000 | < 500ms | 代理人 APP 高峰 |
| 理赔报案 | 100 | < 2000ms | 灾害事件后 |
| 支付回调 | 300 | < 1000ms | 续期扣费日 |
| 续期批处理 | 5000（批处理） | < 30min（全量） | 续期扣费批次 |

### 3.3 资源指标参考

| 资源 | 安全水位 | 告警水位 | 危险水位 |
| --- | --- | --- | --- |
| CPU | < 50% | 50%-70% | > 70% |
| 内存 | < 60% | 60%-80% | > 80% |
| 磁盘 IO | < 40% | 40%-70% | > 70% |
| 网络带宽 | < 40% | 40%-60% | > 60% |
| 数据库连接 | < 60% | 60%-80% | > 80% |
| GC 频率 | — | 每分钟超过 5 次 FGC | FGC 导致暂停 > 1s |

## 4. 压测场景设计

### 4.1 场景模板

```yaml
# 场景名称：[业务场景名称]
# 关联需求：[需求编号]

# 压力配置
threads:
  initial: 10          # 初始并发
  ramp_up: 30         # 爬坡时间（秒）
  peak: 200           # 峰值并发
  duration: 300       # 持续压测时间（秒）

# 接口列表
endpoints:
  - name: "在线投保"
    url: "/v1/underwriting/evaluate"
    method: POST
    weight: 40        # 场景权重（%）
    data:             # 参数化数据源
      source: "proposal_data.csv"
      strategy: "random"

  - name: "保单查询"
    url: "/v1/policy/{policy_no}"
    method: GET
    weight: 60
    data:
      source: "policy_no_list.csv"
      strategy: "sequential"

# 断言
assertions:
  - response_time_p99_lt: 2000
  - error_rate_lt: 0.001
```

### 4.2 必压场景清单

| 序号 | 场景 | 说明 | 峰值 QPS 预估 |
| --- | --- | --- | --- |
| 1 | 投保提交 → 核保 → 出单 | 核心链路 | 500 |
| 2 | 保单查询（按号/按客户） | 高频查询 | 2000 |
| 3 | 理赔报案 + 材料上传 | 文件上传场景 | 100 |
| 4 | 支付回调处理 | 外部系统回调 | 300 |
| 5 | 续期批量扣费 | 批处理场景 | 5000（批处理） |
| 6 | 保全变更 + 重新计算 | 写操作场景 | 200 |
| 7 | 登录 + 权限校验 | 混合场景基础 | 800 |
| 8 | 数据报表查询 | 复杂查询场景 | 50 |

## 5. 压测执行流程

### 5.1 准备阶段

- [ ] 确定压测目标（性能基准 vs 容量极限 vs 稳定性）
- [ ] 搭建压测环境（与生产环境配置等比或按比例）
- [ ] 准备测试数据（满足场景覆盖，数据量接近生产规模）
- [ ] 部署监控（APM / Prometheus / Grafana）
- [ ] 编写压测脚本
- [ ] 执行冒烟测试（验证脚本可用性）

### 5.2 执行阶段

1. **基准测试**：单接口、低并发（1-5 线程），确定基线 RT
2. **负载测试**：逐步增加并发，观察性能拐点
3. **峰值测试**：使用目标峰值压力持续运行 30 分钟
4. **稳定性测试**：70% 峰值压力持续运行 4-8 小时
5. **极限测试**：持续增加压力直至系统崩溃或达到瓶颈

### 5.3 压测停止条件

- 错误率超过 5% 且持续增长
- 核心接口 P99 RT 超过目标值的 3 倍
- CPU / 内存持续超过危险水位
- 数据库主从延迟超过 30 秒
- 监控发现关键服务已熔断/降级

## 6. 结果分析与报告

### 6.1 报告模板

```markdown
# 性能压测报告：[项目名称] v[版本号]

## 测试概要
- 压测日期：YYYY-MM-DD
- 压测环境：[环境信息]
- 压测工具：[工具名称]

## 核心指标汇总

| 场景 | 目标 QPS | 实际 QPS | 目标 P99 | 实际 P99 | 错误率 | 结论 |
| --- | --- | --- | --- | --- | --- | --- |
| 在线投保 | 500 | 520 | 1000ms | 850ms | 0.01% | 通过 |
| 保单查询 | 2000 | 2100 | 500ms | 320ms | 0% | 通过 |

## 资源使用情况

| 服务 | CPU | 内存 | GC | 备注 |
| --- | --- | --- | --- | --- |
| policy-service | 45% | 3.2G/8G | FGC: 0 | 正常 |
| underwriting-service | 52% | 4.1G/8G | FGC: 1 | 正常 |

## 瓶颈与优化建议
1. [瓶颈描述] → [优化建议]
2. [瓶颈描述] → [优化建议]

## 容量预估
- 当前配置支持：[X] QPS
- 未来 6 个月预估需求：[Y] QPS
- 建议扩容：[建议方案]
```

### 6.2 通过/不通过判定

| 判定 | 条件 |
| --- | --- |
| 通过 | 所有核心场景达到目标 QPS 且 P99 RT 和错误率在阈值内 |
| 有条件通过 | 非核心场景未达目标，但有明确的优化计划 |
| 不通过 | 核心场景未达目标，或存在性能瓶颈可能导致生产故障 |
