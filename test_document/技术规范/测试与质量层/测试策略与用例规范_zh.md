# 测试策略与用例规范

> 参考：Google 测试认证体系、阿里巴巴测试规范、ThoughtWorks 测试金字塔、ISTQB 标准

## 1. 概述

### 1.1 目的
规范保险核心系统的测试策略、用例设计和执行流程，确保测试覆盖充分、质量可度量，特别针对核心系统的账务测试和精算接口验证。

### 1.2 适用范围
本规范适用于技术开发部所有产品/项目的功能测试、集成测试、账务测试和精算接口验证。

## 2. 测试金字塔与策略

### 2.1 测试分层策略

| 层 | 比例 | 执行速度 | 职责 | 工具 |
| --- | --- | --- | --- | --- |
| 单元测试（UT） | 60% | 毫秒级 | 验证函数/方法的逻辑正确性 | JUnit / pytest / Go test |
| 集成测试（IT） | 25% | 秒级 | 验证模块间交互、DB 操作、接口调用 | Testcontainers / Mockito |
| 端到端测试（E2E） | 10% | 分钟级 | 验证核心业务流程完整链路 | Selenium / Cypress |
| 手工探索测试 | 5% | N/A | 边界场景、复杂业务逻辑验证 | 人工 |

### 2.2 测试四象限

| 象限 | 测试类型 | 保险业重点关注 |
| --- | --- | --- |
| Q1：面向技术/支持团队 | 单元测试、组件测试 | 保费计算函数、核保规则引擎、再保分摊算法 |
| Q2：面向业务/支持团队 | 功能测试、E2E、UAT | 出单流程、理赔流程、续期扣费、保全变更 |
| Q3：面向业务/评审产品 | 探索测试、场景测试 | 异常流程、边缘案例、监管要求场景 |
| Q4：面向技术/评审产品 | 性能测试、安全测试 | 承保高峰期压测、数据安全合规验证 |

## 3. 用例设计规范

### 3.1 用例模板

```markdown
# TC-[模块]-[序号]：[测试标题]

## 基本信息
- 关联需求：[BRD/TRD 编号]
- 测试类型：[功能/账务/精算/集成/性能]
- 优先级：[P0/P1/P2/P3]
- 自动化：[是/否]

## 前置条件
1. [条件1]

## 测试步骤
1. [步骤1]
2. [步骤2]

## 预期结果
1. [结果1]
2. [结果2]

## 测试数据
| 字段 | 值 | 说明 |
| --- | --- | --- |
| [字段] | [值] | [说明] |

## 清理步骤
[测试完成后的数据清理]
```

### 3.2 用例设计原则

1. **等价类划分** — 每个业务规则的正向/反向/边界值都必须覆盖
2. **边界值分析** — 保费计算、费率区间、免赔额等数值型字段
3. **决策表** — 核保规则的多条件组合（健康告知 + 职业 + 保额 → 决策）
4. **状态转换** — 保单生命周期（投保 → 承保 → 有效 → 理赔 → 终止）
5. **异常场景** — 超时、重试、幂等、并发、数据不一致
6. **数据驱动** — 使用参数化数据覆盖多种产品、渠道、客户类型

## 4. 账务测试规范

### 4.1 账务测试范围

账务测试是保险核心系统测试中最关键的部分，涵盖：

| 账务类型 | 测试要点 |
| --- | --- |
| 保费收取 | 首期保费、续期保费、加费/减费、犹豫期退保 |
| 赔付支出 | 理赔金、满期金、生存金、红利给付 |
| 佣金结算 | 首年佣金、续年佣金、团队管理津贴 |
| 再保结算 | 合同再保分出、临时再保、再保摊回 |
| 准备金 | 未到期责任准备金、未决赔款准备金 |
| 税务处理 | 增值税（保费）、印花税、代扣代缴个税 |

### 4.2 账务验证方法

**方法一：T 型账验证**

针对每笔交易，从会计等式验证资金流向：

```
                    [T型账 - 保费收取]

 借方（应收保费）         贷方（保费收入）
 ┌─────────────────┐  ┌─────────────────┐
 │ 1000.00         │  │ 1000.00         │
 └─────────────────┘  └─────────────────┘
```

**方法二：会计分录验证**

```
交易：收取首期保费 1000 元
借：银行存款                    1000.00
贷：保费收入 - 首期保费         1000.00

交易：支付佣金 200 元
借：手续费及佣金支出             200.00
贷：银行存款                     200.00
```

**方法三：余额核对法**

```
期初余额 + 本期收入 - 本期支出 = 期末余额
   0     +   1000    -   200    =   800
```

### 4.3 账务测试Checklist

- [ ] 单业务场景的账务分录正确
- [ ] 多笔并发交易的账务汇总正确
- [ ] 涉及多币种交易的折算汇率正确
- [ ] 冲正/红字分录正确
- [ ] 跨日、跨月、跨年的账务结转正确
- [ ] 与财务系统对账结果一致

## 5. 精算接口验证

### 5.1 验证范围

| 精算接口 | 验证内容 |
| --- | --- |
| 费率计算接口 | 输入年龄段/性别/保额/缴费年期 → 输出保费 |
| 现金价值接口 | 各保单年度的现金价值、减保后现金价值 |
| 准备金评估接口 | 不同评估标准下的准备金金额 |
| 利润测试接口 | 内含价值、新业务价值 |
| 再保分摊接口 | 自留额、分保额、再保保费 |

### 5.2 验证方法

```markdown
## 验证用例：费率计算接口

### 输入
- 产品代码：PROD-001（终身寿险）
- 年龄：35
- 性别：M
- 保额：1,000,000
- 缴费年期：20年
- 缴费频率：年缴

### 预期输出
- 年缴保费：12,350.00 元（引用精算定价模型输出）
- 月缴保费：1,050.00 元

### 验证规则
- 与精算定价模型输出差异不超过 0.01 元
- 月缴保费 × 12 > 年缴保费（月缴有附加费用）
- 保费与保额成比例增长（线性验证）
```

### 5.3 验收标准

- 费率计算结果与精算模型偏差小于 0.01 元
- 现金价值表 100% 字段匹配精算模型
- 准备金在不同评估标准下的误差在可接受范围内
- 精算接口响应时间在 SLA 范围内（P99 < 2s）

## 6. 测试执行与报告

### 6.1 测试阶段

| 阶段 | 执行者 | 准入条件 | 出口条件 |
| --- | --- | --- | --- |
| 单元测试 | 开发人员 | 代码完成 | 单元测试覆盖率 > 80% |
| 集成测试 | 开发 + QA | UT 通过 | 核心接口测试通过 |
| 系统测试 | QA | IT 通过 | 全部测试用例通过率 > 95% |
| UAT | 业务方 | 系统测试通过 | 关键业务场景验证通过 |
| 回归测试 | QA | 版本变更 | 核心用例 100% 通过 |
